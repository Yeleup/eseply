# Business Rules

Документация является источником правды для бизнес-логики проекта.

## Назначение системы

Система предназначена для учёта коммунальных услуг: абоненты, лицевые счета, услуга организации, типы абонентов, тарифы, счётчики, показания, оплаты, начисления, сальдо, закрытие месяца, перерасчёты и квитанции.

## Терминология

В документации и коде используются следующие соответствия:

| Бизнес-термин | Техническое имя |
|---|---|
| Tenant-организация | `Organization` |
| Абонент коммунальных услуг | `Client` |
| Лицевой счёт | поле клиента `account_number` |
| Услуга организации | `UtilityService` / основная коммунальная услуга tenant-организации |
| Тип абонента | `ClientType`, поле клиента `client_type` |
| Регион | `Region` |
| Улица | `Street` |
| Счётчик | `Meter` |
| Показание счётчика | `MeterReading` |
| Оплата | `Payment` |
| Квитанция | `Receipt` |
| Расчётный месяц | `BillingPeriod` |
| Услуга клиента | услуга выбранной tenant-организации, применяемая к абоненту |

Термин “абонент” в пользовательском интерфейсе и бизнес-правилах соответствует сущности `Client`.

Лицевой счёт хранится как поле клиента, например `account_number`.

Поддерживаемые услуги:

- водоснабжение;
- электроэнергия;
- вывоз мусора;
- канализация;
- обслуживание;
- другие услуги при необходимости.

## Основной принцип расчёта

Основная расчётная связка:

`Лицевой счёт + Расчётный месяц`

Основной расчётный период — месяц.

Расчётный месяц хранится в модели `BillingPeriod` и таблице `billing_periods`.

В учётных таблицах используется связь `billing_period_id`.

Отчёт ошибок закрытия расчётного месяца хранится в таблице `billing_period_closure_errors`.

Код месяца `ГГГГММ` используется только как отображение первого дня месяца `starts_on`.

Доступные статусы расчётного месяца: `open`, `processing`, `closed`, `failed`.

Новый расчётный месяц можно открыть только как следующий календарный месяц после последнего существующего расчётного месяца tenant-организации.

Новый расчётный месяц нельзя открыть, пока последний существующий расчётный месяц не закрыт.

Оператор не вводит код нового расчётного месяца вручную: система автоматически создаёт следующий календарный месяц.

Ввод оплат, показаний счётчиков и корректировок сальдо всегда относится к текущему расчётному месяцу tenant-организации в статусе `open` или `failed`.

Оператор не выбирает расчётный месяц при создании оплат, показаний и корректировок.

Оплаты, показания счётчиков и корректировки сальдо можно менять только в месяцах со статусом `open` или `failed`.

Закрытый месяц нельзя изменять.

Повторное закрытие месяца в статусе `closed` или `processing` запрещено.

Услуга берётся из выбранной tenant-организации.

Тип начисления указывается в карточке абонента.

Внутри одной организации все абоненты обслуживаются по одной коммунальной услуге.

## Организация

В системе хранятся данные организации, от имени которой ведётся учёт.

Организация является tenant-областью Filament: оператор работает в выбранной организации.

У каждой tenant-организации должна быть одна основная коммунальная услуга.

Внутри организации не ведётся набор услуг.

Если нужна новая услуга, создаётся отдельная tenant-организация под эту услугу.

Пользователь может быть участником нескольких организаций.

Пользователь может видеть и открывать только те организации, к которым он привязан.

При создании организации через Filament текущий пользователь автоматически привязывается к этой организации.

Роль пользователя хранится в привязке пользователя к организации, а не глобально на пользователе.

Доступные роли участника организации:

- оператор, техническое значение `operator`;
- контроллер, техническое значение `controller`.

Новый создатель организации получает роль `operator`.

Оператор работает со всеми данными выбранной tenant-организации.

Контроллер может иметь зону ответственности внутри организации по регионам и отдельным улицам.

Назначение регионов и улиц используется только для роли `controller`.

Для ролей, отличных от `controller`, назначения регионов и улиц не сохраняются и не применяются.

Для контроллера списки абонентов, счётчиков, показаний и ведомость снятия показаний ограничиваются назначенными регионами и улицами.

Контроллер может просматривать страницы абонентов и счётчиков только в своей зоне ответственности.

Контроллер не создаёт, не изменяет и не удаляет абонентов.

Контроллер не создаёт, не изменяет, не архивирует и не удаляет счётчики.

Контроллер может вводить и изменять показания счётчиков только по абонентам своей зоны ответственности.

Удаление показаний счётчиков доступно только оператору.

Оплаты, корректировки сальдо, начисления, тарифы, квитанции, адресные справочники, профиль организации и управление пользователями доступны только оператору.

Тип начисления не относится к организации.

Тип начисления указывается только в карточке абонента.

## Адресные справочники

Регионы хранятся в модели `Region` и таблице `regions`.

Улицы хранятся в модели `Street` и таблице `streets`.

Регион относится к tenant-организации.

Улица относится к tenant-организации и региону.

Название региона должно быть уникальным внутри tenant-организации.

Название улицы должно быть уникальным внутри одного региона tenant-организации.

`organization_id` улицы должен совпадать с `organization_id` региона.

Регионы управляются в профиле организации как связанный справочник.

Улицы управляются внутри карточки региона как связанный справочник.

Абонент выбирает регион из справочника текущей tenant-организации.

После выбора региона абонент выбирает улицу только из улиц этого региона.

Улица в карточке абонента должна принадлежать выбранному региону и текущей tenant-организации.

Дом и квартира / помещение в карточке абонента являются необязательными полями.

## Абоненты и лицевые счета

У абонента не выбирается отдельная услуга.

Услуга абонента всегда равна услуге выбранной tenant-организации.

Один абонент имеет один основной лицевой счёт.

Лицевой счёт должен быть уникальным внутри организации.

Лицевой счёт формируется автоматически при создании абонента.

Автоматическое формирование лицевого счёта ведётся отдельно внутри каждой tenant-организации.

Первый автоматически присвоенный лицевой счёт в каждой организации — `100001`; следующие абоненты получают следующий номер по порядку.

Оператор не редактирует лицевой счёт: поле отображается только для чтения.

После создания абонента лицевой счёт не меняется.

Префиксы для физических и юридических лиц не используются.

По лицевому счёту ведутся:

- начисления;
- оплаты;
- сальдо;
- показания;
- квитанции.

Карточка абонента является основным рабочим экраном по лицевому счёту.

На странице редактирования абонента оператор может открыть печатную Blade-карточку абонента.

Печатная Blade-карточка абонента содержит данные абонента, счётчики, показания счётчиков, оплаты, корректировки сальдо, начисления и квитанции. Печатная версия должна быть оптимизирована под A4 и использовать компактные таблицы, чтобы уменьшить расход бумаги.

Внутри карточки абонента отображаются связанные таблицы:

- счётчики;
- оплаты;
- начисления;
- квитанции.

Для контроллера внутри карточки абонента отображаются только счётчики как контекст для ввода показаний.

Показания отображаются внутри карточки конкретного счётчика.

## Услуги клиента

Услуга не выбирается в карточке абонента.

Внутри tenant-организации не ведётся справочник нескольких коммунальных услуг.

Если абоненту нужна другая коммунальная услуга, для этой услуги создаётся отдельная tenant-организация и абонент ведётся в ней отдельно.

Услуга используется при:

- начислении;
- формировании квитанции;
- отчётах.

История начислений сохраняет снимок услуги организации на момент расчёта.

## Типы начисления

Тип начисления выбирается отдельно для каждого абонента.

Тип начисления хранится в карточке клиента в поле `billing_type`.

У одного абонента может быть только один активный тип начисления.

Доступные типы начисления:

- по счётчику, значение `meter`;
- на одного человека, значение `per_person`;
- фиксированная сумма, значение `fixed`.

Если выбран тип “по счётчику”, у абонента должен быть один или несколько активных счётчиков.

Если выбран тип “на одного человека”, используется количество проживающих и тариф для типа абонента.

Если выбран тип “фиксированная сумма”, начисляется указанная сумма без счётчика и тарифа.

Поле `residents_count` является обязательным полем данных абонента, отображается в секции «Данные абонента», имеет значение по умолчанию `1` и не может быть меньше `1`.

В форме абонента поле `fixed_amount` показывается только для `billing_type = fixed`.

Для `billing_type = meter` дополнительные поля настроек начисления не показываются.

На первом этапе параметры начисления хранятся в таблице `clients`:

- `client_type`;
- `billing_type`;
- `residents_count`;
- `fixed_amount`.

Поля `iin`, `contract` и `phone` являются обязательными в форме абонента.

Поля `iin` и `phone` должны быть уникальными внутри tenant-организации. В разных tenant-организациях одинаковые ИИН и телефон допускаются.

Поле `technical_conditions` является nullable текстовым полем карточки абонента.

Поле `utility_service_id` не должно быть пользовательской настройкой абонента.

Если `utility_service_id` технически хранится в связанных таблицах для снимков или связей, его значение должно соответствовать услуге выбранной tenant-организации.

## Тарифы

Тарифы хранятся в модели `Tariff` и таблице `tariffs`.

Тариф относится к tenant-организации, услуге этой организации и типу абонента.

Основные технические поля тарифа:

- `organization_id`;
- `utility_service_id`;
- `client_type`;
- `unit_price`;
- `per_person_price`;
- `starts_on`;
- `status`.

Типы абонентов хранятся в enum `ClientType`.

Доступные типы абонентов:

- Физ. лицо;
- ИП;
- ТОО;
- Коммерческие объекты;
- Бюджет.

Отдельного справочника категорий тарифов нет.

Тип абонента выбирается в карточке клиента.

Тариф выбирается по услуге организации, типу абонента и периоду.

В расчёте используется тариф, действующий на выбранный период.

Для начисления по счётчику используется `unit_price`.

Для начисления на одного человека используется `per_person_price`.

Для фиксированной суммы тариф не используется.

Старые тарифы должны сохраняться для истории расчётов.

## Нормативы

Отдельный справочник нормативов на первом этапе не используется.

Начисление на одного человека считается напрямую по сумме `per_person_price`, указанной в тарифе для типа абонента.

Таблица `normatives`, модель `Normative` и CRUD-раздел нормативов не входят в целевую модель первого этапа.

## Счётчики

Счётчики хранятся в модели `Meter` и таблице `meters`.

Счётчик используется только для абонентов с типом начисления “по счётчику”.

Счётчик относится к tenant-организации и абоненту, а коммунальная услуга берётся из tenant-организации.

Номер счётчика должен быть уникальным внутри tenant-организации.

У абонента может быть несколько активных счётчиков.

Дата снятия счётчика не редактируется вручную в форме.

Снятие счётчика выполняется действием «Отправить в архив» с подтверждением.

При отправке в архив система проставляет `removed_on` текущей датой и переводит `status` в `removed`.

Возврат счётчика в активный учёт выполняется действием «Вывести из архива» с подтверждением.

При выводе из архива система очищает `removed_on` и переводит `status` в `active`.

Начальное показание используется при первом вводе показаний.

Счётчики абонента отображаются и создаются внутри карточки абонента как связанная таблица.

В связанной таблице счётчиков карточки абонента оператор может добавить или изменить показание текущего расчётного месяца сразу для выбранного счётчика.

В связанной таблице счётчиков карточки абонента контроллер может добавить или изменить показание текущего расчётного месяца только по счётчику из своей зоны ответственности.

Создание, редактирование, архивирование, вывод из архива и удаление счётчиков выполняет только оператор.

## Показания счётчиков

Показания счётчиков хранятся в модели `MeterReading` и таблице `meter_readings`.

Показания нужны только для абонентов с типом начисления “по счётчику”.

За один период по одному счётчику должно быть только одно показание.

Основной ввод показаний выполняется внутри карточки счётчика через связанную таблицу показаний или быстрым действием у счётчика в карточке абонента.

Быстрое действие в карточке абонента отображается как «Добавить показание», если в текущем расчётном месяце показания по счётчику ещё нет.

Быстрое действие в карточке абонента отображается как «Изменить показание», если в текущем расчётном месяце показание по счётчику уже есть, и обновляет найденную запись.

При создании показания поле `previous_reading` заполняется системой автоматически.

При первом вводе предыдущее показание берётся из начального показания счётчика.

В следующих месяцах предыдущее показание берётся из прошлого текущего показания.

Если показание создаётся за более поздний период, предыдущее показание берётся из последнего `current_reading` этого счётчика за период раньше выбранного.

Если указана корректировка, в расчёте используется корректированное значение.

Расход считается по формуле:

`Расход = Текущее показание - Предыдущее показание`

Если показания за период нет хотя бы по одному активному счётчику абонента, начисление по счётчику за этот период не создаётся, а закрытие месяца показывает ошибку оператору.

## Оплаты

Оплаты хранятся в модели `Payment` и таблице `payments`.

Оплата привязывается к лицевому счёту.

По одному абоненту может быть несколько оплат за период.

Оплата относится к tenant-организации и абоненту.

Если указана корректировка оплаты, в расчёте используется корректированная сумма.

Оплаты должны отображаться в карточке абонента и в квитанции.

Основной ввод оплат выполняется внутри карточки абонента через связанную таблицу оплат.

При закрытии месяца все оплаты абонента за период суммируются.

Если за текущий расчётный месяц у абонента уже есть квитанция, создание, изменение или удаление оплаты обновляет в квитанции сумму оплат и конечное сальдо.

Оплата не создаёт квитанцию сама; квитанция создаётся при сохранении показания счётчика.

## Расчёт начислений

Расчёт выполняется по каждому абоненту за выбранный период.

Услуга берётся из выбранной tenant-организации.

Тип начисления берётся из карточки абонента.

### Расчёт по счётчику

Используется, если у абонента выбран тип начисления “по счётчику”.

Формула расхода:

`Расход = Текущее показание - Предыдущее показание`

Если есть корректировка:

`Расход = Корректированное показание - Предыдущее показание`

Формула начисления:

`Начисление = Расход × unit_price`

### Расчёт на одного человека

Используется, если у абонента выбран тип начисления “на одного человека”.

`Начисление = residents_count × per_person_price`

### Фиксированное начисление

Используется, если у абонента выбран тип начисления “фиксированная сумма”.

`Начисление = Фиксированная сумма`

При фиксированном начислении счётчик и тариф не используются.

### Начисление по счётчику

Используется, если у абонента выбран тип начисления “по счётчику”.

`Начисление = сумма consumption по всем активным счётчикам абонента × unit_price`

Для расчёта нужны услуга выбранной tenant-организации, тип абонента, один или несколько активных счётчиков по услуге организации, показание за период по каждому активному счётчику и активный тариф.

## Сальдо

Сальдо ведётся по лицевому счёту.

Входящие остатки, ручные исправления, списания и перерасчёты вводятся через корректировки сальдо.

Если предыдущего начисления нет, начальное сальдо периода равно `0`.

Если предыдущее начисление есть, начальное сальдо берётся из конечного сальдо предыдущего месяца.

Корректировки сальдо относятся к конкретному расчётному периоду и суммируются при закрытии месяца.

Формула конечного сальдо:

`Конечное сальдо = Начальное сальдо + Начисление - Оплата + Корректировка`

Если оплат за период несколько, они суммируются.

В сальдо включаются только подтверждённые оплаты из `Payment`.

Неподтверждённые Kaspi QR-заявки из `PaymentTransaction` не уменьшают долг и не считаются оплатой до статуса `completed`.

После подтверждения Kaspi-заявки создаётся одна подтверждённая оплата со способом `kaspi`.

Если оплата больше начисления и долга, появляется переплата.

Если начисление больше оплаты, появляется задолженность.

## Закрытие месяца

Закрытие выполняется за текущий расчётный месяц.

Закрытый месяц считается зафиксированным.

Данные закрытого месяца используются для истории начислений и контроля сальдо.

Один абонент не должен иметь два расчёта за один и тот же период.

Результат закрытия месяца хранится в `Accrual`.

В `Accrual` сохраняются начисления по фиксированной сумме, на одного человека и по счётчику.

Для начисления на одного человека должен существовать активный тариф для типа абонента с заполненным `per_person_price`.

Для начисления по счётчику должны существовать один или несколько активных счётчиков по услуге организации, показание за период по каждому активному счётчику и активный тариф для типа абонента с заполненным `unit_price`.

Сумма оплат за период сохраняется в `Accrual.paid_amount`.

Сумма корректировок сальдо за период сохраняется в `Accrual.adjustment_amount`.

Для первого расчётного месяца начальное сальдо равно `0`.

Для следующих месяцев начальное сальдо берётся из `closing_balance` последнего предыдущего начисления.

Повторное закрытие уже рассчитанного периода не создаёт дубль начисления по тому же абоненту.

Если данные изменились после закрытия, должен выполняться перерасчёт.

Закрытие месяца рассчитывает всех активных абонентов.

Если хотя бы по одному активному абоненту есть ошибка данных, закрытие месяца не создаёт новые начисления ни по одному абоненту.

В этом случае расчётный месяц получает статус `failed`, а оператор видит отчёт ошибок.

Отчёт ошибок содержит лицевой счёт, абонента, тип начисления, код ошибки, текст причины и технический контекст.

При повторном закрытии месяца старый отчёт ошибок очищается и заменяется новым результатом проверки.

Если по активному абоненту не хватает данных, начисление не создаётся, а процесс показывает ошибку по лицевому счёту и абоненту.

## Перерасчёт

Перерасчёт относится ко второму этапу развития системы.

На первом этапе перерасчёт не выполняется автоматически: базовая логика должна стабильно закрывать месяц, создавать начисления, учитывать оплаты и формировать квитанции по показаниям счётчиков.

Перерасчёт выполняется по выбранному периоду.

Основания для перерасчёта:

- изменились показания;
- изменилась сумма оплаты;
- изменилась корректировка;
- изменился тариф;
- изменился тариф на одного человека;
- изменилась фиксированная сумма;
- был изменён тип начисления;
- была обнаружена ошибка в данных.

При перерасчёте система должна:

1. удалить старый расчёт за выбранный период;
2. обновить существующую квитанцию за выбранный период;
3. заново выполнить расчёт;
4. сохранить обновлённую квитанцию.

Квитанция за период должна остаться одной и той же записью.

Оператор должен видеть, что был выполнен перерасчёт.

История перерасчёта должна быть понятна бухгалтерии.

## Квитанции

Квитанции хранятся в модели `Receipt` и таблице `receipts`.

Квитанции отображаются внутри карточки абонента через связанную таблицу квитанций и отдельным пунктом навигации «Квитанции».

Квитанция формируется при сохранении показания счётчика.

Квитанция формируется по одному абоненту за один расчётный период.

Если показание за тот же период изменяется, существующая квитанция обновляется.

Квитанция не хранит обязательную информацию о начислении клиента и не создаётся закрытием месяца из `Accrual`.

Если в текущем расчётном месяце добавляется, изменяется или удаляется оплата, существующая квитанция этого абонента и периода обновляет сохранённые `paid_amount` и `closing_balance`.

Если квитанции за период ещё нет, изменение оплаты не создаёт её.

Квитанция должна содержать услугу выбранной tenant-организации.

Один абонент не может иметь две квитанции за один период внутри одной tenant-организации.

Если у абонента несколько счётчиков за период, квитанция хранит суммарный расход по всем показаниям этого абонента.

Квитанция должна показывать список счётчиков за период: номер счётчика, предыдущее показание, текущее показание, расход, тариф и сумму по строке.

После формирования квитанция должна быть доступна для просмотра и печати.

Оператор может массово печатать выбранные уже сформированные квитанции tenant-организации из раздела «Квитанции».

Оператор может массово печатать все уже сформированные квитанции tenant-организации по текущим фильтрам периода, региона, улицы и контроллера в разделе «Квитанции».

Фильтр контроллера в разделе «Квитанции» ограничивает печать квитанциями абонентов из зоны ответственности выбранного контроллера.

Кнопка массовой печати по фильтру доступна только когда выбран хотя бы один фильтр.

Массовая печать квитанций не создаёт квитанции, не обновляет их и не пересчитывает суммы.

При изменении показаний старая квитанция не удаляется, а обновляется.

Все суммы отображаются в тенге.

## Отчёты

Отчёты относятся ко второму этапу развития системы.

Первый этап считается завершённым, когда стабильно работает базовый вертикальный сценарий: абонент, услуга организации, тариф по типу абонента, счётчик, показание, оплата, квитанция, закрытие месяца и начисление.

Во втором этапе отчёты строятся по сохранённым данным: клиентам, начислениям, оплатам, сальдо и квитанциям.

Отчёты должны строиться только по данным выбранной tenant-организации.

Каждый зарегистрированный отчёт может открываться в детальном режиме и в сводном режиме.

Сводный режим группирует строки отчёта по контроллерам, районам или улицам.

Сводный режим использует те же ограничения tenant-организации, текущего открытого или ошибочного расчётного месяца и доступа контроллера, что и детальный отчёт.

В сводке по контроллерам строка отчёта попадает в строку каждого контроллера, чья зона ответственности подходит абоненту по району или улице.

Если абонент подходит одному контроллеру одновременно по району и улице, внутри строки этого контроллера строка отчёта учитывается один раз.

Если открыт сводный режим, XLSX-выгрузка должна скачивать сводный отчёт по выбранной группировке; если открыт детальный режим, XLSX-выгрузка должна скачивать детальный отчёт.

Ведомость снятия показаний показывает активные счётчики активных абонентов выбранной tenant-организации.

Для контроллера ведомость снятия показаний показывает только активные счётчики активных абонентов из назначенных регионов и улиц.

В ведомости снятия показаний строки сортируются по лицевому счёту и номеру счётчика, поэтому несколько счётчиков одного абонента отображаются соседними строками.

Предыдущее показание в ведомости снятия показаний берётся из последнего сохранённого текущего показания счётчика. Если показаний ещё нет, используется начальное показание счётчика.

Список не снятых показаний показывает активные счётчики активных абонентов с типом начисления `meter`, по которым нет показания за текущий открытый или ошибочный расчётный месяц.

Если текущий расчётный месяц не открыт, список не снятых показаний не показывает строки и сообщает, что месяц нужно открыть.

Для контроллера список не снятых показаний показывает только активные счётчики активных абонентов из назначенных регионов и улиц.

Предыдущее показание в списке не снятых показаний берётся из последнего сохранённого текущего показания счётчика до текущего расчётного месяца. Если показаний ещё нет, используется начальное показание счётчика.

Процент снятия по контроллерам строится за текущий открытый или ошибочный расчётный месяц по пользователям выбранной tenant-организации с ролью `controller`.

Для каждого контроллера знаменатель процента — активные счётчики активных абонентов с типом начисления `meter` в назначенных контроллеру регионах и улицах.

Для каждого контроллера числитель процента — счётчики из его знаменателя, по которым есть показание за текущий расчётный месяц.

Если счётчик попадает к контроллеру одновременно по региону и улице, он учитывается один раз.

Если у контроллера нет назначенной зоны или в зоне нет активных счётчиков, общее количество, снятые показания, не снятые показания и процент равны нулю.

Отчёт «Новые лицевые счета» показывает абонентов выбранной tenant-организации, созданных в текущем открытом или ошибочном расчётном месяце.

Если текущий расчётный месяц не открыт, отчёт «Новые лицевые счета» не показывает строки и сообщает, что месяц нужно открыть.

Для контроллера отчёт «Новые лицевые счета» показывает только абонентов из назначенных регионов и улиц.

Строки отчёта «Новые лицевые счета» сортируются по лицевому счёту и идентификатору абонента.

Отчёт по оплатам показывает оплаты выбранной tenant-организации за текущий открытый или ошибочный расчётный месяц.

Для контроллера отчёт по оплатам показывает только оплаты абонентов из назначенных регионов и улиц.

Отчёт по неоплаченным показывает квитанции текущего открытого или ошибочного расчётного месяца, где сумма начисления больше суммы оплат.

Неоплаченная сумма в отчёте по неоплаченным рассчитывается как `amount - paid_amount`.

Отчёт «Замена/установка счётчика» показывает счётчики, у которых дата установки `installed_on` или дата снятия `removed_on` попадает в текущий открытый или ошибочный расчётный месяц.

Если счётчик снят в текущем периоде, строка считается заменой или снятием.

Отчёт по долгам показывает квитанции текущего открытого или ошибочного расчётного месяца с положительным конечным сальдо `closing_balance`.

Отчёт по потреблениям показывает показания счётчиков текущего открытого или ошибочного расчётного месяца и расход `consumption`.

## Общие требования к понятности

Все действия должны быть понятны оператору и бухгалтерии.
