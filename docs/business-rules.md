# Business Rules

Документация является источником правды для бизнес-логики проекта.

## Назначение системы

Система предназначена для учёта коммунальных услуг: абоненты, лицевые счета, услуга организации, типы абонентов, тарифы, счётчики, показания, оплаты, начисления, сальдо, закрытие месяца, перерасчёты и квитанции.

## Терминология

В документации и коде используются следующие соответствия:

| Бизнес-термин | Техническое имя |
|---|---|
| Tenant-организация | `Organization` |
| Абонент коммунальных услуг | `Client` |
| Лицевой счёт | поле клиента `account_number` |
| Услуга организации | `UtilityService` / основная коммунальная услуга tenant-организации |
| Тип абонента | `ClientType`, поле клиента `client_type` |
| Счётчик | `Meter` |
| Показание счётчика | `MeterReading` |
| Оплата | `Payment` |
| Квитанция | `Receipt` |
| Услуга клиента | услуга выбранной tenant-организации, применяемая к абоненту |

Термин “абонент” в пользовательском интерфейсе и бизнес-правилах соответствует сущности `Client`.

Лицевой счёт хранится как поле клиента, например `account_number`.

Поддерживаемые услуги:

- водоснабжение;
- электроэнергия;
- вывоз мусора;
- канализация;
- обслуживание;
- другие услуги при необходимости.

## Основной принцип расчёта

Основная расчётная связка:

`Лицевой счёт + Период`

Основной расчётный период — месяц.

Период указывается в формате:

`ГГГГММ`

Услуга берётся из выбранной tenant-организации.

Тип начисления указывается в карточке абонента.

Внутри одной организации все абоненты обслуживаются по одной коммунальной услуге.

## Организация

В системе хранятся данные организации, от имени которой ведётся учёт.

Организация является tenant-областью Filament: оператор работает в выбранной организации.

У каждой tenant-организации должна быть одна основная коммунальная услуга.

Внутри организации не ведётся набор услуг.

Если нужна новая услуга, создаётся отдельная tenant-организация под эту услугу.

Пользователь может быть участником нескольких организаций.

Пользователь может видеть и открывать только те организации, к которым он привязан.

При создании организации через Filament текущий пользователь автоматически привязывается к этой организации.

Тип начисления не относится к организации.

Тип начисления указывается только в карточке абонента.

## Абоненты и лицевые счета

У абонента не выбирается отдельная услуга.

Услуга абонента всегда равна услуге выбранной tenant-организации.

Один абонент имеет один основной лицевой счёт.

Лицевой счёт должен быть уникальным внутри организации.

Лицевой счёт может вводиться вручную или формироваться автоматически.

Префиксы для физических и юридических лиц не используются.

По лицевому счёту ведутся:

- начисления;
- оплаты;
- сальдо;
- показания;
- квитанции.

## Услуги клиента

Услуга не выбирается в карточке абонента.

Внутри tenant-организации не ведётся справочник нескольких коммунальных услуг.

Если абоненту нужна другая коммунальная услуга, для этой услуги создаётся отдельная tenant-организация и абонент ведётся в ней отдельно.

Услуга используется при:

- начислении;
- формировании квитанции;
- отчётах.

История начислений сохраняет снимок услуги организации на момент расчёта.

## Типы начисления

Тип начисления выбирается отдельно для каждого абонента.

Тип начисления хранится в карточке клиента в поле `billing_type`.

У одного абонента может быть только один активный тип начисления.

Доступные типы начисления:

- по счётчику, значение `meter`;
- на одного человека, значение `per_person`;
- фиксированная сумма, значение `fixed`.

Если выбран тип “по счётчику”, у абонента должен быть один или несколько активных счётчиков.

Если выбран тип “на одного человека”, используется количество проживающих и тариф для типа абонента.

Если выбран тип “фиксированная сумма”, начисляется указанная сумма без счётчика и тарифа.

На первом этапе параметры начисления хранятся в таблице `clients`:

- `client_type`;
- `billing_type`;
- `residents_count`;
- `area`;
- `fixed_amount`.

Поле `utility_service_id` не должно быть пользовательской настройкой абонента.

Если `utility_service_id` технически хранится в связанных таблицах для снимков или связей, его значение должно соответствовать услуге выбранной tenant-организации.

## Тарифы

Тарифы хранятся в модели `Tariff` и таблице `tariffs`.

Тариф относится к tenant-организации, услуге этой организации и типу абонента.

Основные технические поля тарифа:

- `organization_id`;
- `utility_service_id`;
- `client_type`;
- `unit_price`;
- `per_person_price`;
- `starts_on`;
- `status`.

Типы абонентов хранятся в enum `ClientType`.

Доступные типы абонентов:

- Физ. лицо;
- ИП;
- ТОО;
- Коммерческие объекты;
- Бюджет.

Отдельного справочника категорий тарифов нет.

Тип абонента выбирается в карточке клиента.

Тариф выбирается по услуге организации, типу абонента и периоду.

В расчёте используется тариф, действующий на выбранный период.

Для начисления по счётчику используется `unit_price`.

Для начисления на одного человека используется `per_person_price`.

Для фиксированной суммы тариф не используется.

Старые тарифы должны сохраняться для истории расчётов.

## Нормативы

Отдельный справочник нормативов на первом этапе не используется.

Начисление на одного человека считается напрямую по сумме `per_person_price`, указанной в тарифе для типа абонента.

Таблица `normatives`, модель `Normative` и CRUD-раздел нормативов не входят в целевую модель первого этапа.

## Счётчики

Счётчики хранятся в модели `Meter` и таблице `meters`.

Счётчик используется только для абонентов с типом начисления “по счётчику”.

Счётчик относится к tenant-организации и абоненту, а коммунальная услуга берётся из tenant-организации.

Номер счётчика должен быть уникальным внутри tenant-организации.

У абонента может быть несколько активных счётчиков.

Если счётчик снят, указывается дата снятия.

Начальное показание используется при первом вводе показаний.

## Показания счётчиков

Показания счётчиков хранятся в модели `MeterReading` и таблице `meter_readings`.

Показания нужны только для абонентов с типом начисления “по счётчику”.

За один период по одному счётчику должно быть только одно показание.

Основной ввод показаний выполняется внутри карточки счётчика через связанную таблицу показаний.

При создании показания поле `previous_reading` заполняется системой автоматически.

При первом вводе предыдущее показание берётся из начального показания счётчика.

В следующих месяцах предыдущее показание берётся из прошлого текущего показания.

Если показание создаётся за более поздний период, предыдущее показание берётся из последнего `current_reading` этого счётчика за период раньше выбранного.

Если указана корректировка, в расчёте используется корректированное значение.

Расход считается по формуле:

`Расход = Текущее показание - Предыдущее показание`

Если показания за период нет хотя бы по одному активному счётчику абонента, начисление по счётчику за этот период не создаётся, а закрытие месяца показывает ошибку оператору.

## Оплаты

Оплаты хранятся в модели `Payment` и таблице `payments`.

Оплата привязывается к лицевому счёту.

По одному абоненту может быть несколько оплат за период.

Оплата относится к tenant-организации и абоненту.

Если указана корректировка оплаты, в расчёте используется корректированная сумма.

Оплаты должны отображаться в карточке абонента и в квитанции.

При закрытии месяца все оплаты абонента за период суммируются.

## Расчёт начислений

Расчёт выполняется по каждому абоненту за выбранный период.

Услуга берётся из выбранной tenant-организации.

Тип начисления берётся из карточки абонента.

### Расчёт по счётчику

Используется, если у абонента выбран тип начисления “по счётчику”.

Формула расхода:

`Расход = Текущее показание - Предыдущее показание`

Если есть корректировка:

`Расход = Корректированное показание - Предыдущее показание`

Формула начисления:

`Начисление = Расход × unit_price`

### Расчёт на одного человека

Используется, если у абонента выбран тип начисления “на одного человека”.

`Начисление = residents_count × per_person_price`

### Фиксированное начисление

Используется, если у абонента выбран тип начисления “фиксированная сумма”.

`Начисление = Фиксированная сумма`

При фиксированном начислении счётчик и тариф не используются.

### Начисление по счётчику

Используется, если у абонента выбран тип начисления “по счётчику”.

`Начисление = сумма consumption по всем активным счётчикам абонента × unit_price`

Для расчёта нужны услуга выбранной tenant-организации, тип абонента, один или несколько активных счётчиков по услуге организации, показание за период по каждому активному счётчику и активный тариф.

## Сальдо

Сальдо ведётся по лицевому счёту.

Для первого расчётного месяца используется стартовое сальдо абонента.

Для следующих месяцев используется конечное сальдо предыдущего месяца.

Формула конечного сальдо:

`Конечное сальдо = Начальное сальдо + Начисление - Оплата`

Если оплат за период несколько, они суммируются.

Если оплата больше начисления и долга, появляется переплата.

Если начисление больше оплаты, появляется задолженность.

## Закрытие месяца

Закрытие выполняется за выбранный период.

Закрытый месяц считается зафиксированным.

Данные закрытого месяца используются для квитанций.

Один абонент не должен иметь два расчёта за один и тот же период.

Результат закрытия месяца хранится в `Accrual`.

В `Accrual` сохраняются начисления по фиксированной сумме, на одного человека и по счётчику.

Для начисления на одного человека должен существовать активный тариф для типа абонента с заполненным `per_person_price`.

Для начисления по счётчику должны существовать один или несколько активных счётчиков по услуге организации, показание за период по каждому активному счётчику и активный тариф для типа абонента с заполненным `unit_price`.

Сумма оплат за период сохраняется в `Accrual.paid_amount`.

Для первого расчётного месяца начальное сальдо берётся из `clients.starting_balance`.

Для следующих месяцев начальное сальдо берётся из `closing_balance` последнего предыдущего начисления.

Повторное закрытие уже рассчитанного периода не создаёт дубль начисления по тому же абоненту.

Если данные изменились после закрытия, должен выполняться перерасчёт.

Закрытие месяца рассчитывает всех активных абонентов.

Если по активному абоненту не хватает данных, начисление не создаётся, а процесс показывает ошибку по лицевому счёту и абоненту.

## Перерасчёт

Перерасчёт относится ко второму этапу развития системы.

На первом этапе перерасчёт не выполняется автоматически: базовая логика должна стабильно закрывать месяц, создавать начисления, учитывать оплаты и формировать квитанции.

Перерасчёт выполняется по выбранному периоду.

Основания для перерасчёта:

- изменились показания;
- изменилась сумма оплаты;
- изменилась корректировка;
- изменился тариф;
- изменился тариф на одного человека;
- изменилась фиксированная сумма;
- был изменён тип начисления;
- была обнаружена ошибка в данных.

При перерасчёте система должна:

1. удалить старый расчёт за выбранный период;
2. удалить старую квитанцию за выбранный период;
3. заново выполнить расчёт;
4. сформировать новую квитанцию.

Новая квитанция должна заменить старую.

Оператор должен видеть, что был выполнен перерасчёт.

История перерасчёта должна быть понятна бухгалтерии.

## Квитанции

Квитанции хранятся в модели `Receipt` и таблице `receipts`.

Квитанция формируется после закрытия месяца.

Квитанция формируется по одному абоненту за один расчётный период.

Квитанция формируется по сохранённому `Accrual`, а не пересчитывается заново на лету.

Квитанция должна содержать услугу выбранной tenant-организации.

Один абонент не может иметь две квитанции за один период внутри одной tenant-организации.

Один `Accrual` не может иметь больше одной квитанции.

После формирования квитанция должна быть доступна для просмотра и печати.

При перерасчёте старая квитанция удаляется и создаётся новая.

## Деньги

Все суммы отображаются в тенге.

## Отчёты

Отчёты относятся ко второму этапу развития системы.

Первый этап считается завершённым, когда стабильно работает базовый вертикальный сценарий: абонент, услуга организации, тариф по типу абонента, счётчик, оплата, закрытие месяца, начисление и квитанция.

Во втором этапе отчёты строятся по сохранённым данным: клиентам, начислениям, оплатам, сальдо и квитанциям.

## Общие требования к понятности

Все действия должны быть понятны оператору и бухгалтерии.
