# Changelog

## 2026-06-14

### Changed

- Квитанции больше не создаются из начислений при закрытии месяца: сохранение показания счётчика создаёт квитанцию, а изменение показания обновляет ту же квитанцию за период.
- Создание, изменение и удаление оплаты за текущий расчётный месяц теперь обновляет сумму оплат и конечное сальдо уже созданной квитанции, не создавая новую квитанцию.
- В просмотр и печатную форму квитанции добавлен список счётчиков с предыдущим показанием, текущим показанием, расходом, тарифом и суммой по строке.
- На странице просмотра квитанции добавлена кнопка «Печатать PDF», открывающая печатное представление квитанции в новой вкладке с tenant-доступом только для оператора.
- В разделе «Квитанции» добавлено bulk-действие «Печатать выбранные» для массовой печати уже сформированных квитанций без создания и перерасчёта.
- В разделе «Квитанции» добавлена кнопка «Печатать по фильтру», которая появляется при выбранном фильтре периода и печатает все квитанции текущего tenant за этот период.
- В разделе «Квитанции» фильтр печати расширен: массовая печать по фильтру учитывает период, регион, улицу и контроллера, а кнопка появляется при выборе любого из этих фильтров.
- Быстрое действие показания в карточке абонента теперь отображается как «Изменить показание», если за текущий расчётный месяц уже есть показание по счётчику, и обновляет найденную запись; для нового месяца действие остаётся добавлением.
- ИИН и телефон абонента теперь обязательны к уникальности внутри tenant-организации; в разных организациях одинаковые значения допускаются.

## 2026-06-11

### Added

- Добавлен модуль расчётных месяцев `BillingPeriod` со статусами `open`, `processing`, `closed`, `failed`.
- Добавлен Filament-раздел «Расчётные месяцы» для открытия месяцев внутри tenant-организации.
- Добавлен отчёт ошибок закрытия месяца `billing_period_closure_errors` с просмотром причин в разделе «Расчётные месяцы».

### Changed

- Учётные таблицы начислений, оплат, показаний, корректировок и квитанций переведены со строкового `period` на связь `billing_period_id`.
- Кнопка «Новый расчётный месяц» теперь автоматически создаёт следующий календарный месяц после последнего закрытого месяца tenant-организации.
- Новый расчётный месяц теперь можно открыть только после закрытия последнего существующего месяца tenant-организации; ручной ввод периода и пропуск месяцев запрещены.
- Кнопка «Закрыть месяц» теперь сразу закрывает текущий открытый или ошибочный расчётный месяц без отдельной страницы выбора периода.
- Формы оплат, показаний и корректировок больше не требуют выбора расчётного месяца; система автоматически использует текущий открытый или ошибочный месяц.
- Закрытие месяца больше не создаёт частичные начисления: если хотя бы у одного активного абонента есть ошибка данных, новые начисления не создаются ни по одному абоненту.
- Закрытие месяца блокирует строку периода на время транзакции и запрещает повторное закрытие уже закрытого месяца.
- Оплаты, показания и корректировки нельзя создавать, изменять или удалять в закрытом расчётном месяце.
- Начальное показание счётчика теперь фиксируется при создании и не изменяется при дальнейшем редактировании счётчика.
- Дата установки счётчика в форме создания теперь по умолчанию заполняется текущей датой.
- Повторное создание показания за тот же расчётный месяц теперь показывает понятную ошибку в форме вместо SQL-ошибки уникального индекса.

## 2026-06-05

- Роль участника теперь хранится в привязке пользователя к организации: доступны роли `operator` и `controller`.
- Создатель новой организации автоматически получает роль `operator`.
- Добавлены зоны ответственности контроллера по регионам и отдельным улицам организации.
- Добавлено управление пользователями организации в профиле tenant-организации: прикрепление существующего пользователя, создание пользователя, выбор роли и назначение зон для контроллера.
- Зафиксировано правило: назначения регионов и улиц сохраняются только для роли `controller`, для остальных ролей такие назначения очищаются.
- Для контроллера списки абонентов, счётчиков, показаний и ведомость снятия показаний ограничиваются назначенной зоной.
- Ввод и изменение показаний контроллером ограничены счётчиками абонентов из его зоны ответственности.
- Уточнены права контроллера: он может просматривать только абонентов и счётчики своей зоны как контекст, не выполняет операции с абонентами, счётчиками и другими разделами учёта, а удаление показаний доступно только оператору.

## 2026-06-04

- Добавлена печатная Blade-карточка абонента, открываемая кнопкой «Карточка» со страницы редактирования абонента; карточка содержит данные абонента, счётчики и оплаты.
- Добавлен раздел Filament «Отчёты» с реестром отчётов и универсальной страницей просмотра отчёта.
- Добавлен первый отчёт «Ведомость снятия показаний» по активным счётчикам активных абонентов tenant-организации.
- Зафиксировано правило сортировки ведомости снятия показаний: строки идут по лицевому счёту и номеру счётчика, поэтому несколько счётчиков одного абонента отображаются соседними строками.
- Для отчёта «Ведомость снятия показаний» добавлено скачивание XLSX-файла с тем же tenant-scope, порядком строк и набором колонок, что и на экранной странице отчёта.

## 2026-06-04

### Changed

- Лицевой счёт абонента присваивается автоматически отдельно внутри каждой tenant-организации: первая автогенерация начинается с `100001`, следующие номера увеличиваются на 1.
- Лицевой счёт закреплён как неизменяемое поле после создания; в форме абонента он отображается только для чтения.
- Поле «Количество проживающих» перенесено в данные абонента, стало обязательным для каждого абонента и получает значение `1` по умолчанию.
- В карточку абонента добавлены текстовые поля ИИН (`iin`), договор (`contract`) и тех. условия (`technical_conditions`); ИИН, договор и телефон обязательны в форме абонента.

## 2026-05-29

### Added

- Добавлен адресный справочник: регионы tenant-организации и улицы внутри региона.
- Зафиксирован UI-сценарий: регионы управляются в профиле организации, улицы управляются в карточке региона как связанная таблица.
- Адрес абонента переведён на структурные поля: регион, улица, дом и квартира / помещение; список улиц зависит от выбранного региона.
- Дата снятия счётчика больше не редактируется вручную: счётчик отправляется в архив и выводится из архива через подтверждаемые действия.
- Добавлены корректировки сальдо: входящие остатки, ручные исправления, списания и перерасчёты теперь вводятся отдельными операциями и учитываются при закрытии месяца.
- В карточке абонента у счётчика добавлено быстрое действие «Добавить показание», которое создаёт показание сразу для выбранного счётчика.

### Changed

- Стартовое сальдо убрано из карточки абонента; первый месяц начинается с нулевого начального сальдо, а входящий долг или переплата вводятся как корректировка сальдо за период.

## 2026-05-28

### Changed

- Заменена модель «категорий тарифов» на enum `ClientType`: Физ. лицо, ИП, ТОО, Коммерческие объекты, Бюджет.
- Тарифы теперь задаются по типу абонента и содержат две суммы: `unit_price` для расчёта по счётчику и `per_person_price` для расчёта на одного человека.
- Отдельный справочник нормативов больше не используется на первом этапе: начисление на одного человека считается как `residents_count * per_person_price`.
- Тип начисления `normative` заменён на `per_person` в документации целевой модели.
- Уточнена логика счётчиков: у одного абонента может быть несколько активных счётчиков, а начисление по счётчику суммирует расход по всем активным счётчикам абонента.
- Уточнён ввод показаний: показания создаются внутри карточки счётчика, а предыдущее показание подставляется автоматически из начального показания счётчика или прошлого текущего показания.
- Уточнён основной экран работы с лицевым счётом: счётчики, оплаты, начисления и квитанции отображаются внутри карточки абонента как связанные таблицы.
- Уточнена форма абонента: для начисления на одного человека показывается только количество проживающих, для фиксированного начисления — только фиксированная сумма, для начисления по счётчику дополнительные поля настроек скрыты; поле `area` убрано из целевой модели.

## 2026-05-27

### Changed

- Уточнена модель услуг: tenant-организация ведёт учёт только по одной коммунальной услуге.
- Зафиксировано правило: если нужна новая коммунальная услуга, создаётся отдельная tenant-организация.
- Убрано правило выбора услуги в карточке абонента; услуга абонента наследуется от выбранной tenant-организации.
- Синхронизированы правила для тарифов, нормативов, счётчиков, начислений и квитанций: они используют услугу организации.
- Перенесено управление услугой в профиль tenant-организации; отдельный CRUD-раздел «Услуги» больше не используется.
- Техническое поле `utility_service_id` у клиентов, тарифов, нормативов и счётчиков теперь заполняется из услуги выбранной организации.

## 2026-05-26

### Added

- Добавлено исходное техническое задание системы учёта коммунальных услуг.
- Создан минимальный файл бизнес-правил `docs/business-rules.md`.
- Создана первичная документация модуля услуги клиента `docs/modules/client-services.md`.
- Зафиксировано правило: у одного абонента может быть только одна активная услуга.
- Зафиксировано правило: тип начисления хранится в карточке абонента, а не в организации.
- Подключена документация модуля организаций как tenant-области Filament.
- Зафиксировано правило: пользователь видит только организации, к которым он привязан.
- Уточнена терминология: tenant-организация — `Organization`, абонент коммунальных услуг — `Client`, лицевой счёт — поле клиента `account_number`, услуга клиента — выбранная основная услуга.
- Добавлена документация модуля клиентов `docs/modules/clients.md`.
- Переименована документация услуги клиента в `docs/modules/client-services.md`, чтобы соответствовать термину `Client`.
- Добавлен минимальный модуль клиентов: модель `Client`, таблица `clients`, привязка к `organization_id`, поля карточки клиента и уникальность `account_number` внутри организации.
- Добавлен Filament resource `ClientResource` для управления клиентами в админке выбранной организации.
- Добавлен справочник коммунальных услуг: модель `UtilityService`, таблица `utility_services`, привязка к `organization_id`, поля `name`, `unit_of_measurement`, `status`, `note`.
- Добавлен справочник категорий тарифов: модель `TariffCategory`, таблица `tariff_categories`, привязка к `organization_id`, выбор категории тарифа у клиента через `tariff_category_id`.
- В клиента добавлены настройки начисления первого этапа: `utility_service_id`, `billing_type`, `residents_count`, `fixed_amount`; существующее поле `tariff_category_id` используется для категории тарифа клиента.
- Добавлены тарифы и нормативы: модели `Tariff` и `Normative`, таблицы `tariffs` и `normatives`, связи с организацией, коммунальной услугой и категорией тарифа.
- Добавлены Filament CRUD-разделы в группе «Учёт» в порядке: «Абоненты» (`ClientResource`), «Услуги» (`UtilityServiceResource`), «Категории тарифов» (`TariffCategoryResource`), «Тарифы» (`TariffResource`), «Нормативы» (`NormativeResource`).
- Добавлен первый вертикальный сценарий начислений: модель `Accrual`, таблица `accruals`, закрытие месяца только для активных абонентов с `billing_type = fixed`, просмотр начислений через `AccrualResource`.
- В закрытие месяца добавлен расчёт по нормативу для `billing_type = normative`.
- Добавлены счётчики и показания: модели `Meter` и `MeterReading`, таблицы `meters` и `meter_readings`, уникальность показания по `meter_id + period`, расчёт расхода `current_reading - previous_reading`, Filament resources `MeterResource` и `MeterReadingResource`.
- Добавлены оплаты: модель `Payment`, таблица `payments`, несколько оплат за период, Filament resource `PaymentResource`; закрытие месяца суммирует оплаты периода и считает конечное сальдо как `opening_balance + amount - paid_amount`.
- Закрытие месяца вынесено в отдельную страницу процесса: выбор периода `YYYYMM`, расчёт всех активных абонентов, сохранение `Accrual`, расчёт по счётчику через `consumption * tariff` и вывод ошибок данных по абонентам.
- Добавлены квитанции: модель `Receipt`, таблица `receipts`, автоматическое создание после закрытия месяца из сохранённого `Accrual` без повторного расчёта и Filament resource `ReceiptResource`.
- Зафиксирована граница этапов: перерасчёты и отчёты относятся ко второму этапу после стабилизации базового сценария учёта.
