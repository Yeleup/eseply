# Модуль: Начисления

## Терминология

Начисление в коде называется `Accrual`.

Таблица начислений называется `accruals`.

## Назначение

Модуль хранит результат закрытия расчётного месяца по абоненту.

Сейчас реализован расчёт по фиксированной сумме, по нормативу и по счётчику.

После успешного создания начисления закрытие месяца создаёт квитанцию `Receipt` из сохранённого `Accrual`.

## Основные поля

| Поле | Описание |
|---|---|
| Организация | Tenant-организация, поле `organization_id` |
| Абонент | Клиент, поле `client_id` |
| Услуга | Коммунальная услуга клиента, поле `utility_service_id` |
| Период | Расчётный месяц в формате `ГГГГММ`, поле `period` |
| Лицевой счёт | Снимок лицевого счёта, поле `account_number` |
| Абонент | Снимок имени / наименования клиента, поле `client_name` |
| Название услуги | Снимок названия услуги, поле `utility_service_name` |
| Тип начисления | Тип начисления клиента на момент закрытия, поле `billing_type` |
| Объём | Расход или нормативный объём, поле `volume` |
| Тариф | Цена за единицу, поле `tariff_price` |
| Начислено | Сумма начисления, поле `amount` |
| Оплачено | Сумма оплат за период, поле `paid_amount` |
| Начальное сальдо | Сальдо на начало периода, поле `opening_balance` |
| Конечное сальдо | Сальдо на конец периода, поле `closing_balance` |
| Дата закрытия | Когда выполнено закрытие, поле `closed_at` |

## Правила расчёта

Закрытие месяца рассчитывает всех активных абонентов.

У абонента должна быть выбрана коммунальная услуга.

Для расчёта по нормативу у абонента должна быть выбрана категория тарифа.

Для расчёта по нормативу должны существовать активные `Tariff` и `Normative`, действующие на начало расчётного периода.

Для расчёта по счётчику у абонента должна быть выбрана категория тарифа.

Для расчёта по счётчику должны существовать активный счётчик по услуге клиента, показание за период и активный `Tariff`, действующий на начало расчётного периода.

Формула фиксированного начисления:

`Начисление = fixed_amount`

Формулы начисления по нормативу:

- на человека: `residents_count * normative * tariff`;
- по площади: `area * normative * tariff`;
- на объект: `normative * tariff`.

Формула начисления по счётчику:

`Начисление = consumption * tariff`

Поле `volume` хранит рассчитанный объём до умножения на тариф.

Поле `tariff_price` хранит цену тарифа, использованную при расчёте.

При закрытии месяца все оплаты абонента за период суммируются и сохраняются в `paid_amount`.

Если у абонента есть предыдущее начисление, начальное сальдо берётся из `closing_balance` последнего предыдущего периода.

Если предыдущего начисления нет, начальное сальдо берётся из `clients.starting_balance`.

Конечное сальдо считается так:

`closing_balance = opening_balance + amount - paid_amount`

Один абонент не может иметь два начисления за один период внутри одной tenant-организации.

Повторное закрытие уже рассчитанного периода не создаёт дубликат начисления по тому же абоненту.

Повторное закрытие уже рассчитанного периода не создаёт дубликат квитанции; если квитанции ещё нет, она создаётся из существующего начисления.

Если по активному абоненту не хватает данных, начисление не создаётся, а процесс закрытия месяца возвращает ошибку с лицевым счётом, именем абонента и причиной.

Если для клиента с типом `normative` нет действующего тарифа или норматива, начисление не создаётся, а закрытие месяца считает такую строку ошибкой данных.

Если для клиента с типом `meter` нет активного счётчика, показания за период или действующего тарифа, начисление не создаётся, а закрытие месяца считает такую строку ошибкой данных.

## Административная панель

Начисления отображаются через Filament resource `AccrualResource` внутри выбранной tenant-организации.

В навигации админки раздел называется «Начисления» и расположен после «Нормативы».

Закрытие месяца запускается отдельной страницей «Закрытие месяца» из списка начислений.

Оператор указывает период в формате `ГГГГММ`.

После запуска оператор видит количество активных абонентов, созданных начислений, пропущенных ранее созданных начислений и ошибок данных.

Ошибки показываются списком по лицевому счёту, абоненту и причине.

В списке начислений должны отображаться период, лицевой счёт, абонент, услуга, тип начисления, объём, тариф, начисленная сумма, оплата, начальное сальдо, конечное сальдо и дата закрытия.
