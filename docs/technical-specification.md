# Техническое задание

Система учёта коммунальных услуг

0. Терминология

В документации и коде используются следующие соответствия:

| Бизнес-термин | Техническое имя |
|---|---|
| Tenant-организация | `Organization` |
| Абонент коммунальных услуг | `Client` |
| Лицевой счёт | поле клиента `account_number` |
| Услуга организации | `UtilityService` / основная коммунальная услуга tenant-организации |
| Тип абонента | `ClientType`, поле клиента `client_type` |
| Регион | `Region` |
| Улица | `Street` |
| Счётчик | `Meter` |
| Показание счётчика | `MeterReading` |
| Оплата | `Payment` |
| Квитанция | `Receipt` |
| Расчётный месяц | `BillingPeriod` |
| Услуга клиента | услуга выбранной tenant-организации, применяемая к абоненту |

Термин “абонент” в пользовательском интерфейсе и бизнес-правилах соответствует сущности `Client`.
Лицевой счёт хранится как поле клиента, например `account_number`.
Лицевой счёт уникален внутри tenant-организации и присваивается автоматически при создании абонента: первый автоматический номер в каждой организации — `100001`, следующие номера увеличиваются на 1. После создания лицевой счёт не меняется и отображается оператору только для чтения.

1. Назначение системы
Необходимо разработать систему для учёта коммунальных услуг.
Система должна позволять вести абонентов, услугу организации, типы абонентов, тарифы, счётчики, показания, оплаты, начисления, сальдо, закрытие месяца, перерасчёты и квитанции.
Система должна поддерживать разные виды коммунальных услуг:


водоснабжение;


электроэнергия;


вывоз мусора;


канализация;


обслуживание;


другие услуги при необходимости.


Tenant-организация всегда ведёт учёт только по одной коммунальной услуге.
Например, отдельная организация создаётся для водоснабжения, отдельная организация — для вывоза мусора.
Если нужно вести новую услугу, оператор создаёт новую tenant-организацию под эту услугу.
Внутри одной организации нельзя добавить несколько услуг и нельзя выбирать разные услуги для разных абонентов.

2. Основной принцип системы
Расчёты ведутся по связке:
Лицевой счёт + Расчётный месяц

Расчётный месяц хранится как отдельная сущность `BillingPeriod`.

Учётные таблицы оплат, показаний, начислений, корректировок и квитанций ссылаются на расчётный месяц через `billing_period_id`.

Отчёт ошибок закрытия расчётного месяца хранится в таблице `billing_period_closure_errors`.

Строка отчёта ошибок хранит `billing_period_id`, `organization_id`, `client_id`, лицевой счёт, имя абонента, тип начисления, код ошибки, текст причины и JSON-контекст.

Код месяца в формате `ГГГГММ` является отображением даты начала месяца `starts_on`, а не отдельной колонкой учётных таблиц.

Расчётный месяц имеет статусы `open`, `processing`, `closed`, `failed`.

Новый расчётный месяц открывается строго следующим календарным месяцем после последнего существующего расчётного месяца tenant-организации.

Если последний существующий расчётный месяц не закрыт, следующий расчётный месяц создать нельзя.

При открытии нового расчётного месяца оператор не вводит период вручную: система автоматически создаёт следующий календарный месяц.

Оплаты, показания счётчиков и корректировки сальдо автоматически привязываются к текущему расчётному месяцу tenant-организации в статусе `open` или `failed`.

Оператор не выбирает расчётный месяц при создании оплат, показаний и корректировок.

Закрытый расчётный месяц нельзя изменять.

Услуга берётся из выбранной tenant-организации.
Тип начисления указывается в карточке абонента.
Внутри одной организации все абоненты обслуживаются по одной коммунальной услуге.

выбирать тип начисления для абонента;

вести тарифы;

печатать квитанции;


просматривать историю начислений, оплат, показаний и перерасчётов;


формировать отчёты.



4. Организация
В системе должны храниться данные организации, от имени которой ведётся учёт.
У каждой tenant-организации должна быть одна основная коммунальная услуга.
Внутри организации не ведётся набор услуг.
Если нужна новая услуга, создаётся отдельная tenant-организация под эту услугу.
Тип начисления не относится к организации.
Тип начисления указывается только в карточке абонента.

Правила организации


организация используется как tenant-область административной панели;


организация имеет ровно одну основную коммунальную услугу;


услуга организации обязательна при настройке организации;


внутри одной организации нельзя добавить вторую коммунальную услугу;


для новой коммунальной услуги создаётся новая tenant-организация;


пользователь может быть участником нескольких организаций;


пользователь видит только организации, к которым он привязан;


при создании организации текущий пользователь становится её участником;

роль пользователя хранится в привязке к организации, а не глобально на пользователе;

доступные роли участника организации: оператор и контроллер;

при создании организации текущий пользователь получает роль оператора;

контроллер может иметь зону ответственности по регионам и отдельным улицам внутри организации;

назначение регионов и улиц используется только для роли контроллера;

для ролей, отличных от контроллера, назначения регионов и улиц не сохраняются и не применяются;

для контроллера списки абонентов, счётчиков, показаний и ведомость снятия показаний ограничиваются назначенными регионами и улицами;

контроллер может просматривать страницы абонентов и счётчиков только в своей зоне ответственности;

контроллер не создаёт, не изменяет и не удаляет абонентов;

контроллер не создаёт, не изменяет, не архивирует и не удаляет счётчики;

контроллер может вводить и изменять показания счётчиков только по абонентам своей зоны ответственности;

удаление показаний счётчиков доступно только оператору;

оплаты, корректировки сальдо, начисления, тарифы, квитанции, адресные справочники, профиль организации и управление пользователями доступны только оператору;


данные учёта должны относиться к выбранной организации, если соответствующий модуль поддерживает tenant-изоляцию.


абоненты, тарифы, счётчики, начисления и квитанции используют услугу выбранной организации.

Справочник адресов организации

В профиле организации должны быть доступны регионы как связанный справочник.
Регион в коде называется Region.
Таблица регионов называется regions.
Основные поля региона: id, organization_id, name.
Регион принадлежит tenant-организации.
Название региона должно быть уникальным внутри tenant-организации.

Улицы региона должны быть доступны внутри карточки региона как связанный справочник.
Улица в коде называется Street.
Таблица улиц называется streets.
Основные поля улицы: id, organization_id, region_id, name.
Улица принадлежит tenant-организации и региону.
organization_id улицы должен совпадать с organization_id региона.
Название улицы должно быть уникальным внутри связки tenant-организация + регион.
Абонент выбирает адрес из этого справочника: сначала регион, затем улицу внутри выбранного региона.
В карточке абонента нельзя выбрать улицу из другого региона или другой tenant-организации.

Лицевой счёт присваивается системой при создании абонента. Оператор не вводит и не редактирует лицевой счёт; после создания поле отображается только для чтения.

На первом этапе данные и настройки начисления хранятся в таблице clients.
Поля данных и настроек начисления клиента: client_type, billing_type, residents_count, fixed_amount.
Поле fixed_amount показывается только при billing_type = fixed.
При billing_type = meter дополнительные поля настроек начисления не показываются.
Поле utility_service_id не должно быть пользовательской настройкой абонента.
Если utility_service_id технически хранится в связанных таблицах для снимков или связей, его значение должно соответствовать услуге выбранной tenant-организации.
Доступные значения billing_type: meter, per_person, fixed.
Поле residents_count является обязательным полем данных абонента, отображается в секции данных абонента, имеет значение по умолчанию 1 и не может быть меньше 1.
Поля iin, contract и phone являются обязательными в форме абонента.
Поля iin и phone должны быть уникальными внутри tenant-организации. В разных tenant-организациях одинаковые ИИН и телефон допускаются.
Поле technical_conditions является nullable текстовым полем карточки абонента.
Адрес абонента хранится структурно: region_id, street_id, house, apartment.
Поле region_id обязательно и выбирается из регионов tenant-организации.
Поле street_id обязательно и выбирается только из улиц выбранного региона.
Поля house и apartment необязательные.
Карточка абонента является основным рабочим экраном по лицевому счёту.
На странице редактирования абонента должна быть кнопка «Карточка», открывающая печатную Blade-страницу с данными абонента, его счётчиками и оплатами.
Внутри карточки абонента должны отображаться связанные таблицы: счётчики, оплаты, начисления и квитанции.
Для контроллера внутри карточки абонента отображаются только счётчики как контекст для ввода показаний.
Показания отображаются внутри карточки конкретного счётчика.

6. Лицевой счёт
Лицевой счёт используется для идентификации абонента в системе.
Правила


лицевой счёт должен быть уникальным внутри организации;


лицевой счёт формируется автоматически при создании абонента и не редактируется оператором;


по лицевому счёту ведутся начисления, оплаты, сальдо, показания и квитанции;


один абонент имеет один основной лицевой счёт;


префиксы для физлиц и юрлиц не используются.



7. Услуга организации
В системе не должно быть набора услуг внутри tenant-организации.
Каждая tenant-организация создаётся под одну коммунальную услугу: например, водоснабжение или вывоз мусора.
Если нужно вести ещё одну услугу, создаётся отдельная tenant-организация.
Технически значение услуги может храниться через UtilityService.
Название Service не используется для модели, чтобы не путать коммунальную услугу с сервисными классами Laravel.
UtilityService не должен использоваться как список услуг, который оператор выбирает внутри уже созданной организации.
Внутри организации активна только услуга этой организации.

8. Услуга клиента
У абонента не выбирается отдельная услуга.
Услуга клиента всегда равна услуге выбранной tenant-организации.
Правила


у одного абонента внутри организации может быть только услуга этой организации;


услуга не выбирается в карточке абонента;


услуга используется при начислении, формировании квитанции и отчётов;


если абоненту нужна другая коммунальная услуга, для этой услуги создаётся отдельная tenant-организация и абонент ведётся в ней отдельно;


история начислений сохраняет снимок услуги организации на момент расчёта.

10. Тарифы
Тариф в коде называется Tariff.
Таблица тарифов называется tariffs.
Если utility_service_id хранится у тарифа, он должен совпадать с услугой выбранной tenant-организации.
Основные технические поля тарифа: organization_id, utility_service_id, client_type, unit_price, per_person_price, starts_on, status.
Тип абонента в коде называется ClientType.
ClientType хранится как enum и используется в карточке клиента и тарифе.
Значения ClientType: Физ. лицо, ИП, ТОО, Коммерческие объекты, Бюджет.
Отдельного справочника категорий тарифов нет.
Система должна хранить тарифы по услуге организации и типам абонентов.

11. Нормативы
Отдельный справочник нормативов на первом этапе не используется.
Начисление на одного человека считается напрямую по сумме `per_person_price`, указанной в тарифе для типа абонента.
Таблица `normatives`, модель `Normative` и CRUD-раздел нормативов не входят в целевую модель первого этапа.
Если позже понадобится физическая норма потребления, например `количество проживающих × норма × тариф`, это должно быть вынесено в отдельный этап и отдельное изменение ТЗ.

15. Расчёт начислений
Расчёт выполняется по каждому абоненту за выбранный расчётный месяц.
Расчётная связка
Лицевой счёт + Расчётный месяц
Услуга берётся из выбранной tenant-организации.
Тип начисления берётся из карточки абонента.

15.1. Расчёт по счётчику
Используется, если у абонента выбран тип начисления “по счётчику”.
Формула расхода
Расход = Текущее показание - Предыдущее показание
Если есть корректировка:
Расход = Корректированное показание - Предыдущее показание
Формула начисления
Начисление = Расход × unit_price

15.2. Расчёт на одного человека
Используется, если у абонента выбран тип начисления “на одного человека”.
Формула начисления
Начисление = Количество проживающих × per_person_price

15.3. Фиксированное начисление
Используется, если у абонента выбран тип начисления “фиксированная сумма”.
Начисление = Фиксированная сумма
При фиксированном начислении счётчик и тариф не используются.
Закрытие месяца рассчитывает всех активных абонентов с billing_type = fixed, per_person и meter.
Для фиксированного начисления нужны услуга выбранной tenant-организации и fixed_amount больше 0.
Для начисления на одного человека используется формула:
residents_count * per_person_price.
Для счётчиков используется суммарный расход показаний за период:
Начисление = сумма consumption по всем активным счётчикам абонента * unit_price.
Для billing_type = meter нужны услуга выбранной tenant-организации, тип абонента, один или несколько активных счётчиков по услуге организации, показание за период по каждому активному счётчику и активный тариф.

16. Сальдо
Система должна рассчитывать начальное и конечное сальдо по лицевому счёту.
Начальное сальдо
Если предыдущего начисления нет, начальное сальдо равно 0.
Для следующих месяцев используется конечное сальдо предыдущего месяца.
Входящие остатки, ручные исправления, списания и перерасчёты вводятся через корректировки сальдо за расчётный период.
Конечное сальдо
Конечное сальдо = Начальное сальдо + Начисление - Оплата + Корректировка
Правила


сальдо ведётся по лицевому счёту;


если оплат за период несколько, они суммируются;


если корректировок за период несколько, они суммируются;


если оплата больше начисления и долга, появляется переплата;


если начисление больше оплаты, появляется задолженность.

18. Перерасчёт
Перерасчёт относится ко второму этапу развития системы.
На первом этапе перерасчёт не выполняется автоматически: базовая логика должна стабильно закрывать месяц, создавать начисления, учитывать оплаты и формировать квитанции.
Система должна позволять выполнить перерасчёт за выбранный период.
Основания для перерасчёта


изменились показания;


изменилась сумма оплаты;


изменилась корректировка;


изменился тариф;


изменился тариф на одного человека;


изменилась фиксированная сумма;


был изменён тип начисления;


была обнаружена ошибка в данных.


При перерасчёте система должна


удалить старый расчёт за выбранный период;


удалить старую квитанцию за выбранный период;


заново выполнить расчёт;


сформировать новую квитанцию.


Правила


перерасчёт выполняется по выбранному периоду;


новая квитанция должна заменить старую;


оператор должен видеть, что был выполнен перерасчёт;


история перерасчёта должна быть понятна бухгалтерии.

25. Раздел “Счётчики”
Счётчики абонента должны быть доступны внутри карточки абонента как связанная таблица.
Из связанной таблицы оператор может добавить счётчик, изменить счётчик, добавить показание для выбранного счётчика, отправить счётчик в архив, вывести счётчик из архива и открыть карточку счётчика.
Карточка конкретного счётчика используется для работы с его показаниями, а быстрое действие в карточке абонента позволяет внести новое показание без перехода.
Функции:


просмотр истории счётчика;


просмотр и ввод показаний внутри карточки счётчика;


просмотр активных и архивных счётчиков.



26. Связанная таблица “Показания” в счётчике
Показания должны быть доступны внутри карточки счётчика как связанная таблица.
Основной сценарий оператора — открыть счётчик и внести показание по этому счётчику или добавить показание напрямую из строки счётчика в карточке абонента.
При создании показания поле “Предыдущее показание” уже заполнено системой.
Функции таблицы:


ввод показаний;


просмотр показаний по периоду;


редактирование показаний;


внесение корректировки;


фильтр по периоду;


просмотр истории показаний.



27. Связанная таблица “Оплаты” в абоненте
Оплаты должны быть доступны внутри карточки абонента как связанная таблица.
Основной сценарий оператора — открыть абонента и внести оплату по его лицевому счёту.
Функции таблицы:


ввод оплаты;


просмотр оплат;


редактирование оплаты;


внесение корректировки;


фильтр по периоду;


просмотр истории оплат.



28. Закрытие месяца из раздела “Начисления”
Функции кнопки «Закрыть месяц»:


автоматический выбор текущего расчётного месяца в статусе open или failed;


запуск закрытия текущего месяца;


расчёт всех активных абонентов;


отображение количества рассчитанных абонентов;


отображение ошибок;

отображение ошибок по лицевому счёту, абоненту и причине;


просмотр результата расчёта;


формирование квитанций.



29. Раздел “Квитанции”
Квитанции должны быть доступны внутри карточки абонента как связанная таблица и отдельным пунктом навигации “Квитанции”.
Ручное создание и редактирование квитанций не используется.
Функции таблицы:


фильтр по периоду;


просмотр квитанции;


печать квитанции;


квитанции создаются и обновляются при сохранении показаний счётчиков.

Дополнительная функция:


массовая печать всех уже сформированных квитанций tenant-организации за выбранный расчётный период;


массовая печать выбранных строк из таблицы раздела “Квитанции”;


массовая печать всех квитанций по текущим фильтрам периода, региона, улицы и контроллера в таблице раздела “Квитанции”;


фильтр контроллера ограничивает массовую печать квитанциями абонентов из зоны ответственности выбранного контроллера;


кнопка массовой печати по фильтру отображается только когда выбран хотя бы один фильтр;


массовая печать открывает печатное HTML-представление в новой вкладке и запускает стандартный диалог печати браузера;


массовая печать не создаёт квитанции, не обновляет сохранённые данные и не выполняет перерасчёт.


обновление существующей квитанции после изменения показания.

31. Отчёты
Отчёты относятся ко второму этапу развития системы, после стабилизации базового вертикального сценария.
Система должна предоставлять следующие отчёты:
ОтчётОписаниеСписок абонентовВсе абоненты с лицевыми счетамиАбоненты по организацииВсе абоненты выбранной tenant-организации и её услугиАбоненты по типу начисленияПо счётчику, на одного человека, фиксированная суммаПоказания за периодВсе введённые показанияОплаты за периодВсе оплаты за выбранный месяцНачисления за периодНачисления по абонентамДолжникиАбоненты с задолженностьюПереплатаАбоненты с положительным балансомКвитанцииСписок сформированных квитанцийСводный отчётОбщая сумма начислений, оплат и задолженности за период

Первый реализованный отчёт второго этапа — ведомость снятия показаний.

Ведомость снятия показаний строится по активным счётчикам активных абонентов выбранной tenant-организации.

Для контроллера ведомость снятия показаний строится только по активным счётчикам активных абонентов из назначенных регионов и улиц.

Колонки ведомости снятия показаний: лицевой счёт, ФИО / наименование, адрес, количество проживающих, счётчик, дата установки, предыдущее показание, пустая колонка для нового показания.

Если у абонента несколько счётчиков, они должны отображаться соседними строками за счёт сортировки по лицевому счёту и номеру счётчика.

Предыдущее показание берётся из последнего сохранённого текущего показания счётчика, а при отсутствии показаний — из начального показания счётчика.

Второй реализованный отчёт второго этапа — список не снятых показаний.

Список не снятых показаний строится по активным счётчикам активных абонентов выбранной tenant-организации с типом начисления “по счётчику”, по которым нет показания за текущий открытый или ошибочный расчётный месяц.

Если текущий расчётный месяц не открыт, список не снятых показаний не выводит строки и показывает пустое состояние с объяснением.

Для контроллера список не снятых показаний строится только по активным счётчикам активных абонентов из назначенных регионов и улиц.

Колонки списка не снятых показаний: лицевой счёт, ФИО / наименование, адрес, количество проживающих, счётчик, дата установки, текущий расчётный период, предыдущее показание.

Предыдущее показание для списка не снятых показаний берётся из последнего сохранённого текущего показания счётчика до текущего расчётного месяца, а при отсутствии показаний — из начального показания счётчика.

32. Поиск и фильтры

по ФИО / наименованию;


по региону, улице, дому или квартире / помещению;


по периоду;


по статусу абонента;


по типу начисления;


по типу абонента;


по наличию задолженности;


по наличию счётчика.



33. Общие правила


Основной расчётный период — месяц.


Код периода отображается в формате ГГГГММ, но новый расчётный месяц создаётся системой автоматически.


Основная связка расчёта: лицевой счёт + период.


У каждой tenant-организации может быть только одна активная коммунальная услуга.


У абонента не выбирается отдельная услуга.


Тип начисления хранится в карточке абонента.


Тип начисления не хранится в организации.


У каждого абонента должен быть выбран тип начисления.


По счётчику начисление выполняется на основании показаний.


На одного человека начисление выполняется на основании `residents_count` и `per_person_price` тарифа.


По фиксированной сумме начисление выполняется без счётчика и тарифа.


Все суммы отображаются в тенге.


Закрытый месяц используется для формирования квитанций.


При изменении данных после закрытия выполняется перерасчёт.


Старые тарифы должны сохраняться для истории.


Все действия должны быть понятны оператору и бухгалтерии.

35. Итоговый результат
В результате должна быть разработана система, которая позволяет вести учёт коммунальных услуг:


водоснабжение;


электроэнергия;


вывоз мусора;


канализация;


обслуживание;


другие услуги при необходимости.


Система должна позволять:


регистрировать абонентов;


вести одну услугу в рамках выбранной tenant-организации;


создавать отдельную tenant-организацию для новой коммунальной услуги;


выбирать тип начисления для абонента;


учитывать счётчики;


вводить показания;


вести тарифы по типам абонентов;


вести тарифы;


фиксировать оплаты;


рассчитывать начисления;


рассчитывать сальдо;


закрывать месяц;


выполнять перерасчёты;


формировать и печатать квитанции;


просматривать историю по каждому абоненту.


Система должна быть простой для оператора, понятной для бухгалтерии и гибкой для разных видов коммунальных услуг.
